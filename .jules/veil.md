## 2024-05-18 — Multiple PII & DoS vulnerabilities mitigated
**Vulnerability/Gap:** Lack of rate limiting on API endpoints, missing explicit file validation for type/size bypass, multiple `print` statements leaking user/partner mapped names, stack traces exposed in error handlers, `console.log` statements in JS leaking payload data, missing `X-Frame-Options` header to prevent clickjacking, and large variables in memory not strictly deleted in a `finally` block.
**Root Cause:** Fast prototyping prioritizing feature delivery over defense-in-depth, leading to missing basic security headers, missing memory hygiene, and leaving debug outputs active in a production system handling extremely sensitive data.
**Fix Applied:** Added dictionary-based rate limiting on `/process`. Replaced simple extension check with a `validate_upload` that checks true file size. Replaced missing `X-Frame-Options` header. Added `@app.errorhandler` to catch all exceptions and mask traces. Removed all PII-leaking prints in `app.py`, `core/parsers.py`, and `core/llm_service.py`. Removed `console.log`/`error`/`warn` from JS. Added explicit `del` of `dfs`, `full_df`, and `flashback_df` in a `finally` block in `app.py`.
**Remaining Risk:** Memory-based dictionary rate limiting resets on restart and is local to the worker; a distributed setup would need a Redis-based limiter. PII scrubber uses basic regex and may not catch all PII variations or formatted numbers.
